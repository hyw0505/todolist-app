## 1. 문서 개요

### 목적

본 문서는 개인별 할일 관리 애플리케이션 **todolist-app**의 비즈니스 도메인을 정의한다. 개발팀, 기획팀, 이해관계자가 공통된 언어와 개념으로 소통하기 위한 기준 문서로 활용한다.

### 범위

- 사용자 인증 (회원가입 / 로그인)
- 개인별 할일(Todo) 생성, 조회, 수정, 삭제
- 할일 시작일 / 종료일 관리
- 할일 상태 관리 및 필터링
- 할일 정렬

### 범위 외

- 팀 단위 일정 공유 및 협업 기능
- 알림 / 푸시 기능
- 외부 캘린더 연동

## 2. 도메인 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| 사용자 (User) | 회원가입을 완료하고 시스템에 인증된 개인 |
| 할일 (Todo) | 사용자가 등록한 개별 작업 단위. 시작일과 종료일을 반드시 가진다 |
| 시작일 (Start Date) | 할일을 시작하기로 지정한 날짜 |
| 종료일 (Due Date) | 할일을 완료해야 하는 목표 날짜 |
| 완료 여부 (Completion) | 사용자가 명시적으로 표시한 할일의 달성 결과 (성공 / 실패) |
| 할일 상태 (Todo Status) | 시작일, 종료일, 완료 여부를 기반으로 시스템이 산출하는 상태값 |
| 필터링 (Filtering) | 특정 상태 조건으로 할일 목록을 좁혀보는 기능 |
| 정렬 (Sorting) | 지정한 기준 필드로 할일 목록의 순서를 변경하는 기능 |
| 오늘 날짜 (Today) | 서버 기준 UTC+9(KST) 타임존의 현재 날짜 (시각 아닌 날짜 단위) |
| Access Token | JWT 기반 단기 인증 토큰. API 요청 시 Authorization 헤더에 포함 |
| Refresh Token | Access Token 갱신을 위한 장기 토큰. 서버 측에서 관리 |

## 3. 핵심 도메인 모델

### 3.1 User (사용자)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 사용자 고유 식별자 |
| email | String | 로그인 계정 (고유값) |
| password | String | 암호화된 비밀번호 |
| name | String | 표시 이름 |
| created_at | DateTime | 가입 일시 |

### 3.2 Todo (할일)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 할일 고유 식별자 |
| user_id | UUID | 소유 사용자 (FK) |
| title | String | 할일 제목 |
| description | String | 할일 상세 내용 (선택) |
| start_date | Date | 시작일 (필수) |
| due_date | Date | 종료일 (필수) |
| is_completed | Boolean | 사용자 완료 처리 여부 |
| is_success | Boolean | 성공적 완료 여부 (is_completed=true일 때 유효) |
| status | Enum | 시스템 산출 상태 (§6 참조) |
| created_at | DateTime | 생성 일시 |
| updated_at | DateTime | 최종 수정 일시 |

## 4. 비즈니스 규칙

| ID | 규칙 |
|----|------|
| BR-01 | 할일 관리 기능은 인증된 사용자만 이용할 수 있다 |
| BR-02 | 할일을 생성할 때 시작일과 종료일은 반드시 입력해야 한다 |
| BR-03 | 종료일은 시작일과 같거나 이후 날짜여야 한다 |
| BR-04 | 각 사용자는 본인이 생성한 할일만 조회, 수정, 삭제할 수 있다 |
| BR-05 | 완료 처리(is_completed=true) 시 성공 여부(is_success)를 함께 지정해야 한다 |
| BR-06 | 할일 상태(status)는 시작일, 종료일, 완료 여부를 기반으로 시스템이 자동 산출한다. "오늘 날짜"는 서버 기준 UTC+9(KST) 타임존의 날짜를 사용한다 |
| BR-07 | 정렬 기준은 시작일 또는 종료일 중 하나를 선택할 수 있으며, 오름차순/내림차순을 지원한다 |
| BR-08 | 인증은 JWT 기반 Access Token / Refresh Token 방식으로 처리한다. Access Token 만료 시 Refresh Token으로 재발급하며, Refresh Token이 만료되면 재로그인이 필요하다 |
| BR-09 | OVERDUE 상태의 할일도 사용자가 완료 처리할 수 있다. 이 경우 is_success 값에 따라 COMPLETED_SUCCESS 또는 COMPLETED_FAILURE로 전이된다 |

## 5. 주요 유스케이스

### 5.1 유스케이스 목록

| ID | 유스케이스 | 액터 | 요약 |
|----|-----------|------|------|
| UC-01 | 회원가입 | 비인증 사용자 | 이메일과 비밀번호로 계정을 생성한다 |
| UC-02 | 로그인 | 비인증 사용자 | 이메일/비밀번호 인증 후 Access Token과 Refresh Token을 발급받는다 |
| UC-03 | 할일 생성 | 인증된 사용자 | 제목, 시작일, 종료일을 입력하여 할일을 등록한다 |
| UC-04 | 할일 목록 조회 | 인증된 사용자 | 본인의 할일을 상태 필터 및 정렬 조건으로 조회한다 |
| UC-05 | 할일 수정 | 인증된 사용자 | 제목, 설명, 시작일, 종료일을 수정한다 |
| UC-06 | 할일 완료 처리 | 인증된 사용자 | 할일을 완료 상태로 변경하고 성공/실패 여부를 기록한다 (OVERDUE 상태 포함) |
| UC-07 | 할일 삭제 | 인증된 사용자 | 본인의 할일을 삭제한다 |

### 5.2 유스케이스별 수락 기준 (Acceptance Criteria)

#### UC-01 회원가입

| # | Given | When | Then |
|---|-------|------|------|
| AC-01-1 | 이메일이 시스템에 등록되지 않은 상태 | 유효한 이메일과 비밀번호로 회원가입을 요청하면 | 계정이 생성되고 HTTP 201이 반환된다 |
| AC-01-2 | 동일 이메일이 이미 등록된 상태 | 동일 이메일로 회원가입을 재요청하면 | 중복 오류(HTTP 409)가 반환되고 계정은 생성되지 않는다 |

#### UC-02 로그인

| # | Given | When | Then |
|---|-------|------|------|
| AC-02-1 | 등록된 이메일과 올바른 비밀번호가 주어진 상태 | 로그인을 요청하면 | Access Token과 Refresh Token이 발급되고 HTTP 200이 반환된다 |
| AC-02-2 | 등록된 이메일과 잘못된 비밀번호가 주어진 상태 | 로그인을 요청하면 | 인증 오류(HTTP 401)가 반환되고 토큰은 발급되지 않는다 |

#### UC-03 할일 생성

| # | Given | When | Then |
|---|-------|------|------|
| AC-03-1 | 인증된 사용자가 유효한 시작일과 종료일을 입력한 상태 | 할일 생성을 요청하면 | 할일이 저장되고 초기 status가 자동 산출되어 HTTP 201이 반환된다 |
| AC-03-2 | 종료일이 시작일보다 이전 날짜인 상태 | 할일 생성을 요청하면 | 유효성 오류(HTTP 422)가 반환되고 할일은 저장되지 않는다 |

#### UC-04 할일 목록 조회

| # | Given | When | Then |
|---|-------|------|------|
| AC-04-1 | 인증된 사용자가 특정 상태 필터와 정렬 조건을 지정한 상태 | 할일 목록 조회를 요청하면 | 해당 조건에 부합하는 본인의 할일만 지정 순서로 반환된다 |
| AC-04-2 | 다른 사용자의 할일이 존재하는 상태 | 할일 목록 조회를 요청하면 | 타인의 할일은 결과에 포함되지 않는다 |

#### UC-05 할일 수정

| # | Given | When | Then |
|---|-------|------|------|
| AC-05-1 | 인증된 사용자가 본인 소유 할일의 종료일을 변경하는 상태 | 수정을 요청하면 | 변경 내용이 저장되고 status가 새로운 날짜 기준으로 재산출되어 반환된다 |
| AC-05-2 | 인증된 사용자가 타인 소유 할일을 수정 시도하는 상태 | 수정을 요청하면 | 접근 오류(HTTP 403)가 반환되고 데이터는 변경되지 않는다 |

#### UC-06 할일 완료 처리

| # | Given | When | Then |
|---|-------|------|------|
| AC-06-1 | 인증된 사용자가 IN_PROGRESS 상태의 할일에 is_success=true를 지정한 상태 | 완료 처리를 요청하면 | status가 COMPLETED_SUCCESS로 전이되고 HTTP 200이 반환된다 |
| AC-06-2 | 인증된 사용자가 OVERDUE 상태의 할일에 is_success=false를 지정한 상태 | 완료 처리를 요청하면 | status가 COMPLETED_FAILURE로 전이되고 HTTP 200이 반환된다 |

#### UC-07 할일 삭제

| # | Given | When | Then |
|---|-------|------|------|
| AC-07-1 | 인증된 사용자가 본인 소유 할일을 삭제 시도하는 상태 | 삭제를 요청하면 | 해당 할일이 삭제되고 HTTP 204가 반환된다 |
| AC-07-2 | 인증된 사용자가 타인 소유 할일을 삭제 시도하는 상태 | 삭제를 요청하면 | 접근 오류(HTTP 403)가 반환되고 데이터는 삭제되지 않는다 |

## 6. 할일 상태 정의 및 전이 규칙

### 6.1 상태 정의

| 상태값 | 레이블 | 산출 조건 |
|--------|--------|-----------|
| `NOT_STARTED` | 시작 전 | 오늘 날짜(KST) < 시작일, 미완료 |
| `IN_PROGRESS` | 진행 중 | 시작일 <= 오늘 날짜(KST) <= 종료일, 미완료 |
| `COMPLETED_SUCCESS` | 성공적으로 완료 | is_completed=true AND is_success=true |
| `COMPLETED_FAILURE` | 완료하지 못함 | is_completed=true AND is_success=false |
| `OVERDUE` | 종료됨 | 오늘 날짜(KST) > 종료일, 미완료 |

### 6.2 상태 전이 규칙

```
[생성]
   |
   v
NOT_STARTED  ---(시작일 도래)---> IN_PROGRESS
                                      |
                       +--------------+--------------+
                       |                             |
              (종료일 초과, 미완료)          (사용자 완료 처리)
                       |                             |
                       v                      +------+------+
                   OVERDUE            is_success=true  is_success=false
                       |                      |              |
          (사용자 완료 처리)                   v              v
                       |           COMPLETED_SUCCESS  COMPLETED_FAILURE
            +----------+----------+
            |                     |
    is_success=true       is_success=false
            |                     |
            v                     v
  COMPLETED_SUCCESS       COMPLETED_FAILURE
```

- 상태는 시스템이 조회 시점의 날짜(KST 기준)와 완료 여부를 기준으로 자동 계산한다.
- `COMPLETED_SUCCESS` 또는 `COMPLETED_FAILURE` 상태는 사용자가 완료 처리를 되돌리지 않는 한 변경되지 않는다.
- `NOT_STARTED`, `IN_PROGRESS`, `OVERDUE` 상태는 날짜 변화에 따라 자동으로 전이된다.
- `OVERDUE` 상태에서도 사용자가 완료 처리를 수행할 수 있으며, 오늘 날짜(KST) > 종료일이지만 is_completed=true로 변경한 경우 is_success 값에 따라 `COMPLETED_SUCCESS` 또는 `COMPLETED_FAILURE`로 전이된다.
