## 1. 문서 개요

### 목적

본 문서는 개인별 할일 관리 애플리케이션 **todolist-app**의 비즈니스 도메인을 정의한다. 개발팀, 기획팀, 이해관계자가 공통된 언어와 개념으로 소통하기 위한 기준 문서로 활용한다.

### 범위

- 사용자 인증 (회원가입 / 로그인)
- 개인별 할일 (Todo) 생성, 조회, 수정, 삭제
- 할일 시작일 / 종료일 관리
- 할일 상태 관리 및 필터링
- 할일 정렬
- **다국어 지원 (한국어, 영어, 일본어)**

### 범위 외

- 팀 단위 일정 공유 및 협업 기능
- 알림 / 푸시 기능
- 외부 캘린더 연동

## 2. 도메인 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| 사용자 (User) | 회원가입을 완료하고 시스템에 인증된 개인 |
| 할일 (Todo) | 사용자가 등록한 개별 작업 단위. 시작일과 종료일을 반드시 가진다 |
| 시작일 (Start Date) | 할일을 시작하기로 지정한 날짜 |
| 종료일 (Due Date) | 할일을 완료해야 하는 목표 날짜 |
| 완료 여부 (Completion) | 사용자가 명시적으로 표시한 할일의 달성 결과 (성공 / 실패) |
| 할일 상태 (Todo Status) | 시작일, 종료일, 완료 여부를 기반으로 시스템이 산출하는 상태값 |
| 필터링 (Filtering) | 특정 상태 조건으로 할일 목록을 좁혀보는 기능 |
| 정렬 (Sorting) | 지정한 기준 필드로 할일 목록의 순서를 변경하는 기능 |
| 오늘 날짜 (Today) | 서버 기준 UTC+9(KST) 타임존의 현재 날짜 (시각 아닌 날짜 단위) |
| Access Token | JWT 기반 단기 인증 토큰.API 요청 시 Authorization 헤더에 포함 |
| Refresh Token | Access Token 갱신을 위한 장기 토큰. 서버 측에서 관리 |
| 다국어 지원 (i18n) | 한국어 (ko), 영어 (en), 일본어 (jp) 의 3 개 언어를 지원하는 기능 |

## 3. 핵심 도메인 모델

### 3.1 User (사용자)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 사용자 고유 식별자 |
| email | VARCHAR(255) | 사용자 이메일 (고유) |
| password | VARCHAR(255) | 해시화된 비밀번호 (bcrypt) |
| name | VARCHAR(50) | 사용자 이름 |
| created_at | TIMESTAMP | 회원가입 일시 |
| language | VARCHAR(2) | 사용자 선호 언어 (기본값: ko) |

### 3.2 Todo (할일)

| 속성 | 타입 | 설명 |
|------|------|------|
| id | UUID | 할일 고유 식별자 |
| user_id | UUID | 할일 소유자 (User.id 에 대한 외래 키) |
| title | VARCHAR(100) | 할일 제목 |
| description | TEXT | 할일 설명 (선택) |
| start_date | DATE | 할일 시작 날짜 |
| due_date | DATE | 할일 종료 날짜 |
| is_completed | BOOLEAN | 사용자 명시적 완료 여부 (기본값: false) |
| is_success | BOOLEAN | 완료 시 성공/실패 (null: 미완료, true: 성공, false: 실패) |
| created_at | TIMESTAMP | 생성 일시 |
| updated_at | TIMESTAMP | 수정 일시 |

## 4. 비즈니스 규칙 (Business Rules)

### BR-01. 할일 날짜 제약

- **BR-01-01:** 모든 할일은 시작일과 종료일을 반드시 가져야 한다.
- **BR-01-02:** 종료일은 시작일 이전일 수 없다.

### BR-02. 할일 상태 자동 산출

- **BR-02-01:** 할일의 상태는 시작일, 종료일, 완료 여부를 기반으로 시스템이 자동 계산한다.
- **BR-02-02:** 상태 계산은 서버 기준 KST 날짜를 사용한다.

### BR-03. 완료 처리

- **BR-03-01:** 사용자는 언제든지 할일을 완료 처리할 수 있다.
- **BR-03-02:** 완료 처리 시 `is_completed = true` 로 설정되며, `is_success` 값을 함께 입력한다.

### BR-04. 상태 기반 필터링

- **BR-04-01:** 사용자는 특정 상태로 할일을 필터링할 수 있다.
- **BR-04-02:** 필터링은 여러 상태를 동시에 선택할 수 있다.

### BR-05. 정렬

- **BR-05-01:** 사용자는 임의의 필드로 할일을 정렬할 수 있다.
- **BR-05-02:** 정렬 순서는 오름차순 (asc) 또는 내림차순 (desc) 을 선택할 수 있다.

### BR-06. 데이터 소유권

- **BR-06-01:** 각 할일은 생성자에게만 귀속된다.
- **BR-06-02:** 사용자는 자신의 할일만 조회, 수정, 삭제할 수 있다.

### BR-07. 페이지네이션

- **BR-07-01:** 할일 목록 조회 시 페이지네이션을 지원한다.
- **BR-07-02:** 기본 페이지 크기는 20 이다.

### BR-08. 인증 및 권한 부여

- **BR-08-01:** 모든 API 요청 (공개 엔드포인트 제외) 은 유효한 Access Token 이 필요하다.
- **BR-08-02:** Access Token 은 JWT 를 사용하며, 만료 시 Refresh Token 으로 갱신한다.
- **BR-08-03:** Refresh Token 은 HTTP-only 쿠키로 저장된다.

### BR-09. 다국어 지원

- **BR-09-01:** 시스템은 한국어 (ko), 영어 (en), 일본어 (jp) 를 지원한다.
- **BR-09-02:** 사용자는 프로필에서 선호 언어를 설정할 수 있다.
- **BR-09-03:** 미설정 시 브라우저 Accept-Language 헤더를 사용하며, 기본값은 한국어다.
- **BR-09-04:** 모든 사용자 표시 메시지 (UI, API 응답, 이메일) 는 선택된 언어로 제공된다.

## 5. 유스케이스 (Use Cases)

### UC-01 회원가입

**배경:** 새 사용자가 시스템에 계정을 생성한다.

**전제 조건:** 없음

**기본 흐름:**

1. 사용자가 이메일, 비밀번호, 이름을 입력한다.
2. 시스템은 입력값을 검증한다.
3. 시스템은 이메일 중복을 확인한다.
4. 시스템은 비밀번호를 해시화한다.
5. 시스템은 사용자 데이터를 저장한다.
6. 시스템은 Access Token 과 Refresh Token 을 발급한다.
7. 시스템은 토큰과 사용자 정보를 반환한다.

**대체 흐름:**

- **2a. 이메일 형식 오류:**
  - 시스템은 이메일 형식 오류를 반환한다.
- **3a. 중복 이메일:**
  - 시스템은 resource_conflict 오류를 반환한다.
- **2b. 비밀번호 규칙 위반:**
  - 시스템은 validation_error 를 반환한다.

**수락 기준 (AC):**

- AC-01: 유효한 이메일, 8-64 자 비밀번호, 1-50 자 이름이 필요하다.
- AC-02: 비밀번호는 영문 + 숫자 + 특수문자를 포함해야 한다.
- AC-03: 회원가입 시 자동으로 로그인 상태가 된다.
- **AC-04: 회원가입 성공 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-02 로그인

**배경:** 기존 사용자가 시스템에 인증한다.

**전제 조건:** 사용자가 회원가입을 완료했다.

**기본 흐름:**

1. 사용자가 이메일과 비밀번호를 입력한다.
2. 시스템은 이메일로 사용자를 조회한다.
3. 시스템은 비밀번호를 검증한다.
4. 시스템은 Access Token 과 Refresh Token 을 발급한다.
5. 시스템은 토큰과 사용자 정보를 반환한다.

**대체 흐름:**

- **2a. 존재하지 않는 이메일:**
  - 시스템은 invalid_credentials 오류를 반환한다.
- **3a. 비밀번호 불일치:**
  - 시스템은 invalid_credentials 오류를 반환한다.

**수락 기준 (AC):**

- AC-01: 로그인 성공 시 JWT 토큰 쌍을 발급한다.
- AC-02: Refresh Token 은 http-only 쿠키로 설정된다.
- AC-03: 로그인 실패 시 구체적인 오류 원인을 명시하지 않는다 (보안).
- **AC-04: 로그인 관련 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-03 할일 생성

**배경:** 사용자가 새 할일을 등록한다.

**전제 조건:** 사용자가 로그인했다.

**기본 흐름:**

1. 사용자가 제목, 설명 (선택), 시작일, 종료일을 입력한다.
2. 시스템은 입력값을 검증한다.
3. 시스템은 할일 데이터를 저장한다.
4. 시스템은 생성된 할일을 반환한다.

**대체 흐름:**

- **2a. 제목 누락:**
  - 시스템은 validation_error 를 반환한다.
- **2b. 날짜 형식 오류:**
  - 시스템은 validation_error 를 반환한다.
- **2c. 종료일 < 시작일:**
  - 시스템은 validation_error 를 반환한다.

**수락 기준 (AC):**

- AC-01: 제목은 필수 (1-100 자), 설명은 선택 (0-1000 자) 이다.
- AC-02: 시작일과 종료일은 `YYYY-MM-DD` 형식이다.
- AC-03: 생성 시 `is_completed = false`, `is_success = null` 이다.
- **AC-04: 생성 성공 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-04 할일 목록 조회

**배경:** 사용자가 자신의 할일 목록을 조회한다.

**전제 조건:** 사용자가 로그인했다.

**기본 흐름:**

1. 사용자가 필터 (상태), 정렬 (필드, 순서), 페이지네이션 (page, limit) 옵션을 지정한다.
2. 시스템은 사용자의 할일 목록을 조회한다.
3. 시스템은 각 할일의 상태를 계산한다.
4. 시스템은 필터, 정렬, 페이지네이션을 적용한다.
5. 시스템은 할일 목록과 메타데이터 (total, page, limit) 를 반환한다.

**대체 흐름:**

- **1a. 필터/정렬 없음:**
  - 시스템은 기본값 (필터 없음, created_at 내림차순, page=1, limit=20) 을 사용한다.

**수락 기준 (AC):**

- AC-01: 목록은 사용자의 할일만 포함한다.
- AC-02: 각 할일은 계산된 status 필드를 가진다.
- AC-03: 페이지네이션 메타데이터를 포함한다.
- **AC-04: 응답 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-05 할일 수정

**배경:** 사용자가 자신의 할일을 수정한다.

**전제 조건:** 사용자가 로그인했고, 할일이 존재한다.

**기본 흐름:**

1. 사용자가 수정할 필드 (제목, 설명, 시작일, 종료일) 를 입력한다.
2. 시스템은 입력값을 검증한다.
3. 시스템은 할일을 업데이트한다.
4. 시스템은 수정된 할일을 반환한다.

**대체 흐름:**

- **1a. 수정 필드 없음:**
  - 시스템은 validation_error 를 반환한다.
- **2a. 날짜 관계 위반:**
  - 시스템은 validation_error 를 반환한다.

**수락 기준 (AC):**

- AC-01: 최소 하나의 필드는 수정되어야 한다.
- AC-02: 수정 시 `updated_at` 이 갱신된다.
- AC-03: 완료된 할일도 수정할 수 있다.
- **AC-04: 수정 성공 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-06 할일 완료 처리

**배경:** 사용자가 할일을 완료 처리한다.

**전제 조건:** 사용자가 로그인했고, 할일이 존재한다.

**기본 흐름:**

1. 사용자가 `is_success` (true/false) 를 지정한다.
2. 시스템은 할일을 완료 상태로 업데이트한다.
3. 시스템은 업데이트된 할일을 반환한다.

**대체 흐름:**

- **1a. 이미 완료된 할일:**
  - 시스템은 resource_conflict 오류를 반환한다.
- **1b. is_success 누락:**
  - 시스템은 validation_error 를 반환한다.

**수락 기준 (AC):**

- AC-01: 완료 처리 시 `is_completed = true` 로 설정된다.
- AC-02: `is_success` 는 필수 (boolean) 이다.
- AC-03: 완료 처리 후 상태는 `COMPLETED_SUCCESS` 또는 `COMPLETED_FAILURE` 가 된다.
- **AC-04: 완료 처리 메시지는 사용자의 선호 언어로 제공된다.**

---

### UC-07 할일 삭제

**배경:** 사용자가 자신의 할일을 삭제한다.

**전제 조건:** 사용자가 로그인했고, 할일이 존재한다.

**기본 흐름:**

1. 사용자가 할일 삭제를 요청한다.
2. 시스템은 할일을 삭제한다.
3. 시스템은 204 No Content 를 반환한다.

**대체 흐름:**

- **1a. 존재하지 않는 할일:**
  - 시스템은 404 not_found 를 반환한다.

**수락 기준 (AC):**

- AC-01: 삭제된 할일은 복구할 수 없다.
- AC-02: 삭제 성공 시 204 No Content 를 반환한다.
- **AC-03: 삭제 관련 메시지는 사용자의 선호 언어로 제공된다.**

## 6. 할일 상태 전이 규칙 (State Transition Rules)

### 6.1 상태 정의

| 상태 | 조건 |
|------|------|
| NOT_STARTED | 현재 날짜 < 시작일, `is_completed = false` |
| IN_PROGRESS | 시작일 ≤ 현재 날짜 ≤ 종료일, `is_completed = false` |
| OVERDUE | 현재 날짜 > 종료일, `is_completed = false` |
| COMPLETED_SUCCESS | `is_completed = true`, `is_success = true` |
| COMPLETED_FAILURE | `is_completed = true`, `is_success = false` |

### 6.2 상태 전이 규칙

```
NOT_STARTED  ---(시작일 도래)---> IN_PROGRESS
                                      |
                       +--------------+--------------+
                       |                             |
              (종료일 초과, 미완료)          (사용자 완료 처리)
                       |                             |
                       v                      +------+------+
                   OVERDUE            is_success=true  is_success=false
                       |                      |              |
          (사용자 완료 처리)                   v              v
                       |           COMPLETED_SUCCESS  COMPLETED_FAILURE
            +----------+----------+
            |                     |
    is_success=true       is_success=false
            |                     |
            v                     v
  COMPLETED_SUCCESS       COMPLETED_FAILURE
```

- 상태는 시스템이 조회 시점의 날짜 (KST 기준) 와 완료 여부를 기준으로 자동 계산한다.
- `COMPLETED_SUCCESS` 또는 `COMPLETED_FAILURE` 상태는 사용자가 완료 처리를 되돌리지 않는 한 변경되지 않는다.
- `NOT_STARTED`, `IN_PROGRESS`, `OVERDUE` 상태는 날짜 변화에 따라 자동으로 전이된다.
- `OVERDUE` 상태에서도 사용자가 완료 처리를 수행할 수 있으며, 오늘 날짜 (KST) > 종료일이지만 is_completed=true 로 변경한 경우 is_success 값에 따라 `COMPLETED_SUCCESS` 또는 `COMPLETED_FAILURE`로 전이된다.
