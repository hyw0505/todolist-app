## 1. 문서 개요

### 1.1 목적

본 문서는 todolist-app의 주요 기능을 실제 사용자 관점에서 시나리오로 기술한다. 개발팀과 QA팀이 사용자 흐름을 구체적으로 이해하고, 기능 구현 및 테스트의 기준으로 활용하는 것을 목적으로 한다.

### 1.2 참조 문서

| 문서명 | 경로 |
|--------|------|
| 도메인 정의서 | `docs/1-domain-definition.md` |
| PRD | `docs/2-prd.md` |

### 1.3 커버리지

본 문서의 시나리오는 UC-01(회원가입), UC-02(로그인), UC-03(할일 생성), UC-04(할일 목록 조회), UC-05(할일 수정), UC-06(할일 완료 처리), UC-07(할일 삭제) 전체를 포함한다.

## 2. 페르소나 정의

### P-01. 이지은 (17세, 고등학생)

- **특성:** 스마트폰 위주 사용, 앱 설치보다 모바일 웹 선호, 디지털 서비스 경험 풍부
- **목적:** 수행평가, 시험 범위, 과제 제출일 등 학업 일정을 한눈에 파악
- **주요 불편:** 마감일이 지난 과제를 놓쳐도 기록이 남지 않아 반성이 어려움
- **사용 환경:** 모바일 웹 (iOS Safari)
- **선호 언어:** 한국어 (ko)

### P-02. 박민준 (32세, 직장인)

- **특성:** 데스크탑과 모바일을 병행, 업무 효율에 민감, 새로운 도구를 빠르게 습득
- **목적:** 여러 프로젝트 일정과 개인 업무를 시작일·종료일 기준으로 체계적으로 관리
- **주요 불편:** 기존 메모앱은 날짜 기반 필터링 기능이 없어 우선순위 파악이 어려움
- **사용 환경:** 데스크탑 (Chrome) 및 모바일 웹 병행
- **선호 언어:** 영어 (en)

### P-03. 김수정 (44세, 자영업자)

- **특성:** 디지털 기기 친숙도 보통, 복잡한 기능보다 단순명확한 UI 선호
- **목적:** 가게 발주, 청소, 세금 신고 등 반복 업무의 마감일 관리
- **주요 불편:** 완료하지 못한 일과 완료한 일이 섞여서 뭘 먼저 해야 할지 모름
- **사용 환경:** 모바일 웹 (Android Chrome)
- **선호 언어:** 한국어 (ko)

### P-04. 정현우 (51세, 공무원)

- **특성:** PC 중심 사용, 새로운 서비스 가입에 신중, 개인정보 보안에 민감
- **목적:** 문서 제출, 회의 준비, 보고서 마감 등 공직 업무 일정 관리
- **주요 불편:** 종료일이 지난 업무를 완료 처리할 방법이 없어 기록이 누락됨
- **사용 환경:** 데스크탑 (Edge)
- **선호 언어:** 일본어 (jp)

### SC-01. 신규 사용자 회원가입 및 첫 할일 등록

**관련 UC:** UC-01(회원가입), UC-02(로그인), UC-03(할일 생성)  
**페르소나:** P-01 이지은 (17세, 고등학생)

#### 상황

이지은은 다음 주에 수학 수행평가와 영어 에세이 제출이 겹쳐 있다. 친구가 알려준 todolist-app을 처음 방문해서 계정을 만들고 두 가지 할일을 등록하려 한다.

#### 정상 흐름 (Happy Path)

| 단계 | 이지은의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | 브라우저에서 서비스 URL 접속 | 로그인/회원가입 화면 표시 (브라우저 `Accept-Language: ko` 감지 → 한국어로 표시) |
| 2 | "회원가입" 버튼 탭 | 회원가입 폼 표시 (이름, 이메일, 비밀번호 입력란) |
| 3 | 이름: "이지은", 이메일: "jieun@example.com", 비밀번호: "Jieun2026!" 입력 | 입력값 실시간 유효성 표시 |
| 4 | "가입하기" 버튼 탭 | 계정 생성 완료 (HTTP 201), "회원가입이 완료되었습니다" 메시지 한국어로 표시, 로그인 화면으로 이동 |
| 5 | 이메일과 비밀번호로 로그인 | Access Token(15분) + Refresh Token(7일) 발급 (HTTP 200), 할일 목록 화면으로 이동 |
| 6 | "할일 추가" 버튼 탭 | 할일 생성 폼 표시 |
| 7 | 제목: "수학 수행평가 제출", 시작일: 2026-04-01, 종료일: 2026-04-07 입력 후 저장 | 할일 저장 (HTTP 201), 오늘(4월 1일)이 시작일과 같으므로 status = IN_PROGRESS로 자동 산출 |
| 8 | 다시 "할일 추가" 버튼 탭 | 할일 생성 폼 표시 |
| 9 | 제목: "영어 에세이 제출", 시작일: 2026-04-05, 종료일: 2026-04-09 입력 후 저장 | 할일 저장 (HTTP 201), 오늘(4월 1일)이 시작일보다 이전이므로 status = NOT_STARTED로 자동 산출 |
| 10 | 할일 목록 화면에서 2개 항목 확인 | "수학 수행평가 제출(진행 중)", "영어 에세이 제출(시작 전)" 표시 |

#### 결과

이지은은 계정 생성 후 즉시 두 개의 할일을 등록했고, 각각의 상태가 자동으로 '진행 중'과 '시작 전'으로 구분되어 표시됨을 확인한다.

### SC-02. 진행 중인 할일 목록 확인 및 정렬/필터링

**관련 UC:** UC-04(할일 목록 조회)  
**페르소나:** P-02 박민준 (32세, 직장인)

#### 상황

박민준은 여러 프로젝트의 할일을 등록해 두었다. 출근 후 오전에 오늘 집중해야 할 진행 중인 할일을 확인하고, 종료일이 빠른 순서로 정렬해 우선순위를 파악하려 한다.

#### 정상 흐름 (Happy Path)

| 단계 | 박민준의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | 앱에 접속 (기존 로그인 상태, Access Token 유효) | 할일 목록 화면 표시 |
| 2 | 상태 필터에서 "진행 중(IN_PROGRESS)" 선택 | 현재 날짜 기준 IN_PROGRESS 상태의 본인 할일만 필터링하여 표시 |
| 3 | 정렬 기준: "종료일", 정렬 방향: "오름차순" 선택 | 종료일이 가장 빠른 항목부터 순서대로 목록 재정렬 |
| 4 | 목록 스크롤하며 항목 확인 | 페이지네이션 적용, 다음 페이지 로드 가능 |
| 5 | 다른 사용자의 할일은 표시되지 않음을 확인 | 본인(user_id 기준) 할일만 반환 (BR-04) |

#### 예외 흐름 - Access Token 만료 시

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 필터 변경 요청 | Access Token 만료 감지 |
| 2 | 자동 처리 (사용자 개입 없음) | Refresh Token으로 /auth/refresh 요청 → 새 Access Token 발급 |
| 3 | 필터 요청 재시도 | 정상적으로 필터링 결과 반환 |

#### 결과

박민준은 종료일 오름차순으로 정렬된 진행 중인 할일 목록을 통해 오늘 가장 먼저 처리해야 할 업무를 빠르게 파악한다.

### SC-03. 할일 수정 및 완료 처리 - 성공

**관련 UC:** UC-05(할일 수정), UC-06(할일 완료 처리)  
**페르소나:** P-02 박민준 (32세, 직장인)

#### 상황

박민준이 등록해 둔 "분기 보고서 작성" 할일의 종료일을 하루 연장해야 한다. 이후 작업을 마치고 해당 할일을 성공 완료 처리한다.

#### 정상 흐름 - 할일 수정 (Happy Path)

| 단계 | 박민준의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | 할일 목록에서 "분기 보고서 작성" 항목 선택 | 할일 상세 화면 표시 |
| 2 | "수정" 버튼 클릭 | 수정 폼 표시 (기존 값 자동 입력) |
| 3 | 종료일을 2026-04-05에서 2026-04-06으로 변경 후 저장 | 변경 내용 저장 (HTTP 200), 새 종료일 기준으로 status 재산출 → IN_PROGRESS 유지 |
| 4 | 수정된 종료일과 상태 확인 | 변경된 값이 화면에 반영됨 |

#### 정상 흐름 - 완료 처리 성공 (Happy Path)

| 단계 | 박민준의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | "분기 보고서 작성" 할일에서 "완료 처리" 버튼 클릭 | 완료 유형 선택 UI 표시 (성공/실패) |
| 2 | "성공적으로 완료" 선택 (is_success = true) | status = COMPLETED_SUCCESS로 전이 (HTTP 200) |
| 3 | 목록에서 해당 항목 상태 확인 | "성공적으로 완료"로 표시됨 |
| 4 | 동일 항목에서 완료 처리 재시도 | 이미 완료된 할일이므로 재완료 불가 안내 |

#### 결과

박민준은 종료일을 조정한 후 작업을 마무리하고 성공 완료 처리까지 완료한다. 이미 완료된 할일은 재완료할 수 없음을 확인한다.

### SC-04. 기한이 지난 할일(OVERDUE) 완료 처리 - 실패 처리

**관련 UC:** UC-06(할일 완료 처리), BR-09  
**페르소나:** P-04 정현우 (51세, 공무원)

#### 상황

정현우는 "민원 처리 보고서 제출" 할일의 종료일이 지났음을 뒤늦게 확인했다. 업무는 실제로 처리하지 못했지만, 기록 목적으로 "완료하지 못함" 처리를 하려 한다. OVERDUE 상태에서도 완료 처리가 가능한지 확인하는 시나리오다.

#### 정상 흐름 (Happy Path - OVERDUE 경로)

| 단계 | 정현우의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | 할일 목록에서 상태 필터 없이 전체 조회 | 모든 상태의 할일 목록 표시 |
| 2 | "민원 처리 보고서 제출" 항목의 상태 확인 | 오늘(KST) > 종료일, 미완료 → status = OVERDUE(종료됨)로 표시됨 |
| 3 | 해당 항목에서 "완료 처리" 버튼 클릭 | 완료 유형 선택 UI 표시 (OVERDUE 상태이지만 완료 처리 가능, BR-09) |
| 4 | "완료하지 못함" 선택 (is_success = false) | status = COMPLETED_FAILURE로 전이 (HTTP 200) |
| 5 | 목록에서 해당 항목 상태 확인 | "완료하지 못함"으로 표시됨 |

#### 결과

정현우는 기한이 지난 할일도 완료 처리할 수 있음을 확인하고, "완료하지 못함" 상태로 기록을 남긴다. OVERDUE 상태의 할일이 BR-09 규칙에 따라 COMPLETED_FAILURE로 정상 전이됨을 확인한다.

### SC-05. 할일 삭제

**관련 UC:** UC-07(할일 삭제)  
**페르소나:** P-03 김수정 (44세, 자영업자)

#### 상황

김수정은 실수로 같은 내용의 할일을 두 번 등록했다. 중복 항목을 삭제하려 한다.

#### 정상 흐름 (Happy Path)

| 단계 | 김수정의 행동 | 시스템 응답 |
|------|-------------|------------|
| 1 | 할일 목록에서 중복 항목 확인 | 동일 제목의 할일 2개 표시 |
| 2 | 삭제할 항목을 선택 후 "삭제" 버튼 탭 | 삭제 확인 다이얼로그 표시 |
| 3 | 확인 버튼 탭 | 해당 할일 삭제 (HTTP 204), 목록에서 항목 제거 |
| 4 | 목록 확인 | 남은 동일 제목 항목 1개만 표시 |

#### 예외 흐름 - 타인의 할일 삭제 시도

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 타인의 할일 ID로 삭제 요청 (직접 API 호출 등) | 권한 오류 반환 (HTTP 403) |
| 2 | 데이터 상태 | 해당 할일은 삭제되지 않음 (BR-04) |

#### 결과

김수정은 중복 등록된 할일을 정상적으로 삭제하고 목록을 정리한다.

#### EX-01. 중복 이메일 회원가입 시도

**관련 UC:** UC-01  
**페르소나:** P-03 김수정 (44세, 자영업자)

**상황:** 김수정이 이미 가입한 이메일로 다시 회원가입을 시도한다.

#### EX-02. 비밀번호 정책 미충족 회원가입 시도

**관련 UC:** UC-01  
**페르소나:** P-01 이지은 (17세, 고등학생)

**상황:** 이지은이 비밀번호 정책을 모르고 단순한 비밀번호("jieun123")로 가입하려 한다.

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 비밀번호: "jieun123" 입력 후 가입 시도 | 입력값 검증 실패 (HTTP 400 Bad Request) |
| 2 | 화면 안내 확인 | 비밀번호 정책 안내 표시 (대문자, 특수문자 포함 필요) |

> 비밀번호 정책: 8자 이상, 대문자 1자 이상, 소문자 1자 이상, 숫자 1자 이상, 특수문자 1자 이상, 공백 불가.

#### EX-03. 잘못된 비밀번호로 로그인 시도

**관련 UC:** UC-02  
**페르소나:** P-04 정현우 (51세, 공무원)

**상황:** 정현우가 비밀번호를 잘못 입력하여 로그인에 실패한다.

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 올바른 이메일 + 잘못된 비밀번호로 로그인 시도 | 인증 오류 (HTTP 401 Unauthorized) |
| 2 | 화면 안내 확인 | "이메일 또는 비밀번호가 올바르지 않습니다" 메시지 표시 |
| 3 | 토큰 발급 여부 확인 | Access Token, Refresh Token 미발급 |

#### EX-04. 종료일이 시작일보다 이전인 할일 생성 시도

**관련 UC:** UC-03, BR-03  
**페르소나:** P-02 박민준 (32세, 직장인)

**상황:** 박민준이 실수로 종료일을 시작일보다 이전 날짜로 입력한다.

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 시작일: 2026-04-10, 종료일: 2026-04-05로 입력 후 저장 시도 | 유효성 오류 (HTTP 422 Unprocessable Entity) |
| 2 | 화면 안내 확인 | "종료일은 시작일과 같거나 이후 날짜여야 합니다" 메시지 표시 |
| 3 | 할일 저장 여부 확인 | 할일 저장되지 않음 |

#### EX-06. Refresh Token 만료로 재로그인 필요

**관련 UC:** UC-02, BR-08  
**페르소나:** P-03 김수정 (44세, 자영업자)

**상황:** 김수정이 7일 이상 앱을 사용하지 않아 Refresh Token이 만료된 상태에서 접속한다.

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | 앱 접속 및 할일 목록 조회 시도 | Access Token 만료 감지 → /auth/refresh 요청 |
| 2 | 자동 처리 | Refresh Token도 만료됨 → 재로그인 요청 응답 (HTTP 401) |
| 3 | 로그인 화면 이동 | "세션이 만료되었습니다. 다시 로그인해 주세요" 안내 표시 |
| 4 | 김수정이 이메일/비밀번호 재입력 | 새 Access Token + Refresh Token 발급, 정상 서비스 재개 |

#### EX-07. 이미 완료된 할일 재완료 시도

**관련 UC:** UC-06  
**페르소나:** P-04 정현우 (51세, 공무원)

**상황:** 정현우가 COMPLETED_SUCCESS 상태의 할일에 다시 완료 처리를 시도한다.

| 단계 | 행동 | 시스템 응답 |
|------|------|------------|
| 1 | COMPLETED_SUCCESS 상태 항목에서 "완료 처리" 재시도 | 재완료 불가 안내 표시 |
| 2 | 상태 변경 여부 확인 | 기존 COMPLETED_SUCCESS 상태 유지, 변경되지 않음 |
